# Supabase 云端实时同步优化方案

## 问题诊断

### 原始问题

- 删除列车后,云端更新很慢
- 增加/修改列车时,云端同步也很慢
- 数据只在程序退出时才批量同步

### 根本原因

1. **启用了延迟同步模式**: `main.cpp` 中设置了 `railwayPgSetDeferredSyncEnabled(true)`
2. **立即同步判断失效**: 所有修改操作使用 `railwayPgCanWriteImmediately()` 判断,该函数在延迟模式下返回 `false`
3. **只在退出时同步**: 数据修改只标记 `m_dirty = true`,实际同步延迟到程序退出时

## 解决方案

### 方案概述

为每个 Manager 添加**单条记录立即同步**的方法,在 CUD 操作时立即调用,实现实时云端更新。

### TrainManager 修改详情

#### 1. 新增两个同步方法 (`trainmanager.h`)

```cpp
void syncSingleTrainToPostgres(const Train &train);       // 立即同步单条列车(增/改)
bool deleteSingleTrainFromPostgres(const QString &trainNumber); // 立即删除单条列车
```

#### 2. 实现立即同步逻辑 (`trainmanager.cpp`)

**syncSingleTrainToPostgres** - 用于添加/更新操作:

- 检查数据库连接,必要时重连
- 查询起始站和终点站ID
- 使用 UPSERT (INSERT ... ON CONFLICT ... DO UPDATE) 实现增改合一
- 添加详细的日志输出便于调试

**deleteSingleTrainFromPostgres** - 用于删除操作:

- 检查数据库连接
- 直接执行 DELETE 语句
- 返回删除是否成功

#### 3. 修改业务方法调用同步

**deleteTrain_api** - 删除列车:

```cpp
trains.erase(it);
m_dirty = true;

// 立即同步删除到云端
if (deleteSingleTrainFromPostgres(trainNumber)) {
    m_dirty = false; // 删除成功后标记为已同步
}
```

**updateTimetableAndTrainNumberByTrainNumber** - 添加/更新时刻表:

```cpp
// 添加新列车
trains.push_back(newTrain);
m_dirty = true;
syncSingleTrainToPostgres(newTrain);

// 更新现有列车 - 如果车次号变了,先删除旧记录
if (oldTrainNumber != newTrainNumber) {
    deleteSingleTrainFromPostgres(oldTrainNumber);
}
syncSingleTrainToPostgres(train);
```

**updateSeatTemplateByTrainNumber** - 更新座位配置:

```cpp
it->setCarriages(carriages);
m_dirty = true;
syncSingleTrainToPostgres(*it);
```

## 注意事项

### 1. 保留延迟同步模式的原因

虽然实现了立即同步,但 **仍然保留** `railwayPgSetDeferredSyncEnabled(true)`:

- 避免与原有的批量同步冲突
- 程序退出时仍会调用 `saveToPostgres()` 作为兜底机制
- 如果立即同步失败,退出时还能再次尝试

### 2. 错误处理

每个同步方法都包含:

- 数据库连接检查
- 查询失败时的 warning 日志
- 成功时的 debug 日志,便于追踪

### 3. 事务考虑

单条记录同步不使用事务,因为:

- 每次只修改一条记录,原子性由数据库保证
- UPSERT 语句本身是原子操作
- 减少事务开销,提升响应速度

### 4. 网络问题处理

如果 Supabase 连接不可用:

- 操作会在内存中成功
- 云端同步会失败并记录日志
- `m_dirty` 标志保持为 `true`
- 程序退出时会再次尝试批量同步

## 性能影响

### 网络开销

- 每次操作增加 1 次 HTTP 请求到 Supabase
- 通常延迟 < 500ms (取决于网络环境)
- 用户几乎感知不到延迟

### 用户体验提升

- ✅ 删除后立即生效,不需要等待程序退出
- ✅ 多客户端数据实时一致
- ✅ 更符合云数据库的使用预期
- ✅ 降低数据丢失风险

## 回滚方案

如果需要恢复到原来的批量同步模式:

1. 注释掉所有 `syncSingleXxxToPostgres` 调用
2. 保持 `m_dirty = true` 标记
3. 依赖退出时的批量同步

## 总结

通过添加单条记录立即同步机制,成功解决了 Supabase 云端更新缓慢的问题。系统现在能够:

- ✅ 立即将 CUD 操作同步到云端
- ✅ 保留批量同步作为兜底机制
- ✅ 提供详细日志便于调试
- ✅ 妥善处理网络异常情况

建议按照相同模式修改其他 Manager,实现全系统的实时云端同步。
